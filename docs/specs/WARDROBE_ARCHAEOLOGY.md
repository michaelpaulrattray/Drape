# Wardrobe archaeology — the residue map for pass 2

**Status: MAP ONLY** (founder-directed at the VC-R5 close-out, 2026-07-12). Wardrobe is acknowledged archaeology but **explicitly out of R6 scope** — its reckoning is pass 2's wardrobe-on-canvas redesign, and restyling it earlier pays for a wall pass 2 rebuilds. This doc lists the known residue as of R5's close so pass 2's planning starts with the map, not a dig. It is a vantage-point inventory, not a fresh audit — pass 2 re-audits before building.

**The framing law (Group 6i):** all of this descends from the dead linear pipeline (cast → wardrobe → export as one conveyor). The decomposition is ratified — casting = a summonable environment producing identity; **wardrobe = a tool/node for dressing references**; export = a verb on the asset. Anything below that assumes the old sequence is belt-plumbing.

## Known residue (by seam)

1. **Its own undo system** — `useWardrobeStore` carries a parallel VTO history (`history`/`historyIndex`, `undoVTO`, per-index selection snapshots + overlay caches, max-entry trimming; `client/src/features/wardrobe/stores/useWardrobeStore.ts` ~L36–285). Same *shape* as casting's now-retired session undo but a separate implementation. D-53's ledger-governs-versions model has no wardrobe counterpart yet — pass 2 decides whether VTO outputs get a ledger of their own (board `InputSnapshot`/version rails are the likely home once wardrobe is a node).
2. **Session-handoff persistence** — the `wardrobe sessions` table (modelId, lastResultUrl, iterationCount, savedLookCount, activeGarmentIds, history, historyIndex, tattooMapData, styleNotes; read via `getRecentUserSessions`). Sessions are the conveyor's memory: "resume where the belt stopped." On-canvas wardrobe replaces session resumption with board persistence (nodes + provenance) — the table's fate (migrate, freeze, or feed) is a pass-2 planning question.
3. **Legacy deep-link routing** — `/studio?tool=wardrobe&modelId=…` (used by `LibraryView` minted cards; the A2(a) chooser keeps routing to it as-is per the R6 boundary). The route embodies "next stage" thinking; pass 2 replaces the destination with a dressing node that receives a cast via edge.
4. **The `/studio` shell seam** — R6's shell-unification assessment must state explicitly what happens to wardrobe's route: inherits the clean environment chrome, or is carved out visibly until pass 2. (Binding note in Group 6i.)
5. **Warm legacy skin** — wardrobe never received the canvas-language pass; its surfaces are pre-`DESIGN_SYSTEM.md` styling throughout. Deliberately untouched by R6's restyle.
6. **Pre-canvas stores** — wardrobe's zustand stores are D-24-class (load-bearing for the current tool, forbidden imports for canvas code). Pass 2's node model decides which state moves to `board_items.metadata` + `useGenerationJobs` per foundations Decision 4.
7. **Namespace inversion worth untangling** — shared plumbing lives under wardrobe's roof: `server/wardrobe/utils` exports `getAiClient`/`withTextQueue`/`toInlinePart`, which CASTING imports (backViewGate, editClassifier), and the wardrobe `identityCheck` was the gate pattern casting generalized. The dependency arrow points the wrong way for a pass-2 world where wardrobe is just another consumer — relocate the shared utils to a neutral module when pass 2 touches these files.
8. **Assets already pass-2-ready** — garment auto-captioning shipped (`server/wardrobe/garmentAnalysis.ts`, audit N4); `library_garment` provenance + `vto_input_model`/`vto_input_garment` edge relations (input class, D-50.5) reserved in the schema; the D-30 composer's reference-budget rules (D-39d) already contemplate multi-model + garment payloads. The picker's garments tab is stubbed (DS §7.3).

## What pass 2 inherits ready-made

The canvas side of wardrobe's future is largely built: typed input pins (`ConnectionDot` prompt/image), input-class edges, `InputSnapshot` provenance, the composer (`composeIdentityPayload` — dressing is its first real consumer), plan/execute + named refunds, job store tolerant of long generations, and the comp card as the identity reference wardrobe consumes. The pass-2 design question is therefore mostly *subtraction*: which of the residue above survives as node internals, and which dies with the conveyor.
